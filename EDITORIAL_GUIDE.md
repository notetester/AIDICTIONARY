# AIDICTIONARY 편집 기준

## 목표 독자

처음에는 용어를 모르는 비전공자일 수 있지만, 설명을 끝까지 읽으면 실무 문서와 논문을 해석할 수 있는 수준에 도달하도록 작성합니다.

## 필수 구조

### 1. 쉬운 설명

- 비유는 개념의 방향을 잡는 데만 사용합니다.
- 비유를 실제 정의처럼 단정하지 않습니다.
- 중요한 예외를 숨기지 않습니다.

### 2. 정확한 정의

- 무엇인지뿐 아니라 적용 조건과 범위를 명시합니다.
- 구현마다 달라질 수 있는 사항은 “일반적으로”, “전형적인 구현에서는”처럼 한계를 표시합니다.
- 서로 다른 개념을 같은 말로 뭉개지 않습니다.
  - 파라미터와 하이퍼파라미터
  - JSON 유효성과 의미 유효성
  - 학습 양자화와 배포 양자화
  - 사실성과 grounding

### 3. 작동 원리

입력 → 중간 처리 → 출력 순서로 설명합니다. 수식이 필요하면 기호의 차원과 의미를 함께 적습니다.

### 4. 트레이드오프

“좋다/나쁘다”가 아니라 다음 축을 분리합니다.

- 품질
- 계산량
- 메모리
- 지연시간
- 구현 복잡성
- 실패 가능성
- 일반화와 과적합

### 5. 흔한 오해

실제 대화와 실험에서 반복되는 잘못된 해석을 구체적으로 적습니다. 단순한 퀴즈형 문장은 피합니다.

### 6. 실무 검증

설정을 적용했다는 주장보다 실제 적용 여부와 결과를 검증하는 방법을 적습니다.

예:

- target module 이름을 출력해 실제 주입 확인
- seed별 raw result 보존
- production parity와 stress test 분리
- 병합·양자화 전후 회귀 평가

## 관계 작성

각 용어는 가능한 경우 다음 관계를 가집니다.

- `prerequisites`: 먼저 이해해야 할 개념
- `related`: 같은 문제를 다른 각도에서 설명하는 개념
- `next`: 현재 개념에서 자연스럽게 확장되는 개념

관계는 단순 태그가 아니라 학습 순서를 구성해야 합니다. 순환 참조는 허용하지만 모든 노드가 자기 자신만 반복하도록 만들지 않습니다.

## 출처 원칙

- 알고리즘과 모델 기법은 원 논문 또는 공식 명세를 우선합니다.
- 라이브러리별 동작은 공식 문서를 기준으로 하고 버전에 따라 달라질 수 있음을 표시합니다.
- 출처는 정의를 대신하지 않습니다. 본문만 읽어도 개념이 완결되어야 합니다.

## 금지 사항

- “무조건”, “완벽히”, “환각을 제거한다” 같은 과도한 단정
- 모델 크기나 단일 지표만으로 품질을 결론내리기
- 통계적 유의성과 실무적 중요성을 혼동하기
- 형식 준수와 내용 정확성을 혼동하기
- 쉬운 설명을 위해 기술적으로 틀린 원인을 제시하기
